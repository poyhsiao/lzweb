Virtual Server

IPv4 Rule Setting This table allows you to configure virtual server, real server and Log for each rule.
Virtual Server
WAN 1
NoneNo virtual server is connected to this WAN link.
AutoThe virtual server will be automatically assigned an IPv4 address in this WAN link.
When this WAN link is enabled in [System/Network Setting/WAN Setting/Basic Subnet], XteraLink will assign an IPv4 address based on WAN link type. e.g.
Static: The first IP address defined in [System/Network Setting/WAN Setting/IPv4 Public IP Passthrough/IP Address/Range] will be assigned.
PPPoE: The first IP address obtained from PPPoE will be assigned.
DHCP: An IP address via DHCP will be assigned.
IPv4 AddressEnter an IPv4 address for the virtual server in this WAN link. e.g. 192.168.1.10
WAN 2Settings same as WAN 1.
WAN 3Settings same as WAN 1.
Service
AnyVirtual server will use any service.
proto@Virtual server will use service packets categorized by protocol number. e.g. proto@1 indicates ICMP packets.
TCP@Virtual server will use service packets categorized by TCP port number. e.g. TCP@80 indicates HTTP packets.
UDP@Virtual server will use service packets categorized by UDP port number. e.g. UDP@53 indicates DNS packets.
group@Virtual server will use packets that match rules defined in [System/Service Group]. e.g. group@number(name)
Real Server
IPv4Enter an IPv4 address for this real server in DMZ or the subnet in LAN.
Port MappingSet up a corresponding port number for the real server.
Log
Unchecked (Disable)The record of this rule being used will not be logged in the page of [Log/View/Traffic].
Checked (Enable)The record of this rule being used will be logged in the page of [Log/View/Traffic].

Notes:

  1. When [Virtual Server/Service] is set as Any/proto@/group@, [Real Server/Port Mapping] will be unavailable for configuration, indicating the virtual server uses the same service as the real server.

  2. When [Virtual Server/Service] is set as TCP@, [Real Server/Port Mapping] must be configured. e.g.
    Single port mapping: Both [Virtual Server/Service] and [Real Server/Port Mapping] use TCP@80 (TCP＠22 mapped to TCP@80)
    Single port change: [Virtual Server/Service] uses TCP@22, while [Real Server/Port Mapping] uses TCP@23 (TCP@22 changed to TCP@23)
    Multiple port mapping: [Virtual Server/Service] uses TCP@22-25, while [Real Server/Port Mapping] uses TCP@22-25 (TCP＠22 mapped to TCP@22, TCP＠23 mapped to TCP@23, and so on.)
    Multiple port change: [Virtual Server/Service] uses TCP@22-25, while [Real Server/Port Mapping] uses TCP@32-35 (TCP＠22 changed to TCP@32, TCP＠23 changed to TCP@33, and so on.)

  3. When [Virtual Server/Service] is set as UDP@, [Real Server/Port Mapping] must be configured. e.g.
    Single port mapping: Both [Virtual Server/Service] and [Real Server/Port Mapping] use UDP@80 (UDP＠22 mapped to UDP@80)
    Single port change: [Virtual Server/Service] uses UDP@22, while [Real Server/Port Mapping] uses UDP@23 (UDP@22 changed to UDP@23)
    Multiple port mapping: [Virtual Server/Service] uses UDP@22-25, while [Real Server/Port Mapping] uses UDP@22-25 (UDP＠22 mapped to UDP@22，UDP＠23 mapped to UDP@23, and so on.)
    Multiple port change: [Virtual Server/Service] uses UDP@22-25, while [Real Server/Port Mapping] uses UDP@32-35 (UDP＠22 changed to UDP@32, UDP＠23 changed to UDP@33, and so on.)

  4. TCP and UDP ports cannot be mapped or changed to each other. Port mapping and change can only be among TCP or UDP ports.

  5. When multiple port mapping or multiple port change is used, the amount of ports used for [Virtual Server/Service] and [Real Server/Port Mapping] must be the same.
